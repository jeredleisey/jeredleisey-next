This project walks through building a repeatable pipeline for turning raw completion reports into structured well summaries.

By the end you'll have a prompt template, a QA checklist, and a clear mental model for when Claude needs more context versus when the prompt is just wrong.

## The Raw Input

Completion reports vary wildly between operators and even between wells in the same program. You might get a 40-page PDF with a clean structure, or a six-page Word document where the DST results are buried in a paragraph labeled "Observations." The common thread is that all the information is there — it's just not organized for someone who needs a quick read on completion quality, IP, and any anomalies.

The goal here wasn't to replace the engineer's judgment. It was to pull the consistent facts out of inconsistent documents so the engineer spends their time thinking, not scanning.

## The Prompt Template Structure

The template that worked best follows a three-part structure: context, instruction, format.

**Context** tells Claude what role it's in and what kind of document it's reading. Without this, you get generic summaries that miss E&P-specific detail. Specificity here matters: "upstream completion report" is better than "technical document."

**Instruction** tells Claude exactly what to extract and in what order. Vague instructions like "summarize the well" produce vague output. Explicit extraction targets — treating zone, perforated interval, proppant loading, 30-day IP, notable problems — produce consistent, structured output.

**Format** locks the output shape. If your downstream use is a Spotfire dashboard or a shared spreadsheet, you need the same fields in the same order every time. Specifying this in the prompt is far more reliable than hoping Claude guesses your schema.

Here's a stripped-down version of the skeleton:

```text
You are an upstream petroleum engineer reviewing a well completion report.

Extract the following information exactly as labeled:
- Well name and API number
- Operator
- Spud date and completion date
- Formation / treating zone
- Perforated interval (MD and TVD)
- Treating fluid type and total volume
- Proppant type and total mass
- ISIP and closure pressure (if reported)
- 30-day IP (oil, gas, water with units)
- Any recorded issues or anomalies during completion

If a field is not present in the report, write "not reported" — do not infer or estimate.

Return results as a labeled list using the exact field names above.

[PASTE REPORT TEXT BELOW]
```

The "not reported" instruction is load-bearing. Without it, Claude will sometimes interpolate from context or omit the field silently, and you won't know you're missing data until it matters.

## QA Checklist

After you run a batch through this prompt, spot-check against these before you trust the output:

- Does every summary have the same fields? Missing fields usually mean the source document used non-standard labeling.
- Do the treating volumes look physically plausible for the zone? Claude will faithfully transcribe a typo in the source.
- Are units consistent? A report might list fluid volume in barrels in one place and gallons in another. Claude will reflect the inconsistency, not resolve it.
- Is the IP realistic for the play? Again, Claude transcribes; it doesn't sanity-check.
- Are anomalies captured, or does the summary read clean when you know there were screen-out issues?

This checklist is mostly about catching source document problems, not Claude problems. Once your prompt is dialed in, Claude's extraction is reliable. The failures usually trace back to scanned PDFs with OCR artifacts, incomplete reports, or data that was never in the document to begin with.

## Pitfalls

**Over-long context windows without structure.** Pasting a 40-page PDF as a single text block and asking for a summary produces worse results than chunking by section and extracting from each chunk. The completion data is almost always in the last 20 pages — start there.

**Assuming Claude knows your play conventions.** "Normal" proppant loadings differ between the Permian, Haynesville, and DJ. If you want Claude to flag something as anomalous, you have to tell it what the expected range is. It doesn't have your basin context unless you give it.

**Using the same prompt for very different document types.** A prompt tuned on AFE completion packets doesn't work well on state regulatory filings. Keep separate templates for meaningfully different source formats.

## The Outcome

With a stable template and a short QA checklist, a single engineer can process a multi-well program's completion reports in an afternoon rather than a week. The summaries aren't prose — they're structured extractions — but that's the point. You get consistent data you can sort, compare, and plug into your existing tools without reformatting.

The more important gain is repeatability. A new team member can run the same pipeline on the same documents and get the same output. That's harder to put a number on, but it matters more than the time savings.
