Every drilling program produces daily reports. They're usually long, inconsistently formatted, and written for compliance as much as communication. By the time they're reviewed in the morning meeting, half the room hasn't read them. This project built a prompt workflow that converts the previous day's reports into a short, skimmable digest — the kind that gets read.

## The Problem with Daily Reports as Written

A contractor daily drilling report covers everything: bit runs, mud properties, personnel on location, fuel consumption, footage drilled, BHA details, NPT events, and safety observations. That completeness is valuable for the record. It's not valuable for a 7 a.m. meeting where you need to know whether you're on depth, on schedule, and whether anything went sideways overnight.

The signal-to-noise ratio is low. The format changes between contractors. And when you're running multiple rigs, you're reading three or four of these documents before the meeting starts — or you're not reading them at all.

## What the Digest Needs to Answer

Before writing a single prompt, it helps to be explicit about what questions the morning meeting actually asks:

- Where are we in the well? (Current MD, formation, last casing point)
- What happened in the last 24 hours? (Footage drilled, any bit or BHA changes)
- Did anything go wrong or slow us down? (NPT, kicks, stuck pipe, lost circulation)
- Are we on track? (Footage vs. plan, any upcoming hazard zones)
- Anything the operator needs to act on today?

That's the scope of the digest. Everything else stays in the full report.

## The Prompt Structure

The prompt follows the same three-part pattern that works for any structured extraction: context, instruction, format.

```text
You are a drilling engineer reviewing a daily drilling report for an operator.
Your job is to produce a concise morning brief for the operations team.

From the attached report, extract and summarize the following:

CURRENT STATUS
- Current measured depth (MD)
- Current formation / section
- Hole section (e.g., 9-7/8" intermediate)

LAST 24 HOURS
- Footage drilled
- Bit/BHA changes (if any)
- Key operations completed

NPT AND ISSUES
- Any non-productive time events, with cause and duration
- Any wellbore integrity concerns (kicks, losses, stuck pipe)
- Nothing to report if none

UPCOMING
- Next planned operation
- Any known hazard zones or expected formation changes in the next 24-48 hours

ACTION ITEMS
- Anything requiring operator decision or approval today

Keep each section to 2-3 sentences or a short bulleted list.
If a field is not addressed in the report, write "not reported."

[PASTE REPORT TEXT BELOW]
```

The NPT section is the most important one to get right. NPT events in daily reports are sometimes buried in narrative, sometimes logged as a code with a short description, sometimes omitted if the contractor is managing optics. The prompt asks for cause and duration explicitly, which forces Claude to look for both — and surfaces when one of them is missing.

## Running It Across Multiple Rigs

When you're running more than one rig, you want a single digest that covers all of them, not one digest per report. The simplest approach: run the per-report prompt on each document first, then pass all the outputs into a second prompt that combines them.

```text
Below are morning briefs for three active rigs on this program.
Produce a combined one-page summary in the same format, with each rig clearly labeled.
Flag any cross-rig issues or patterns worth noting (e.g., similar NPT events on multiple rigs).

[PASTE RIG 1 BRIEF]
[PASTE RIG 2 BRIEF]
[PASTE RIG 3 BRIEF]
```

The two-step approach is more reliable than pasting three full reports into one prompt. You're not asking Claude to track which detail belongs to which rig across a very long context — you're giving it already-structured input and asking it to organize that.

## Pitfalls

**NPT narratives written to minimize.** Contractors sometimes describe a stuck pipe event as "additional reaming required due to formation characteristics." Claude will extract what's written. If you want it to flag that as a potential NPT event, you need to tell it what to look for — or review the digest against the raw report when something feels off.

**Depth units.** Reports mix MD and TVD, and some contractors use both without labeling which is which in context. Add an instruction asking Claude to note which depth convention it's using if it's ambiguous.

**Missing the overnight shift.** Daily reports often cover a 6 a.m. to 6 a.m. window, but the cutoff varies. If your digest is for a 7 a.m. meeting and the report only goes to 6 a.m., you're always missing the last hour. That's a process issue, not a Claude issue — but it's worth noting in the digest so people don't assume it's current.

## The Outcome

With this workflow in place, the morning meeting starts from a shared baseline. Everyone in the room has read the same one-pager — because it's short enough to actually read. The full reports are still there for reference and for the record. But the questions "where are we" and "what happened" are answered before the meeting starts, not during it.

The bigger shift is that the digest creates a written record of what the team knew each morning. When something goes wrong later in the well, that context matters.
